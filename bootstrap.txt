Build an accurate working model of the project in the order below BEFORE proposing or making changes.

WORKING RULES (from the project's own standards)
- Provider quirks belong in that provider's driver/filtering module; shared lifecycle belongs in server/runner. Never let one provider's fix distort another driver.
- Verify state transitions (paste verified, submit verified, candidate-based extraction with logged rejection reasons); never assume.
- Every bug fixed should leave behind a regression assertion or better diagnostic.
- Short valid answers must not be rejected; prefer smallest valid child over parent containers; static status text (e.g. "Stopped generating") is not an active control.
- Update Documentation/Reference/11 (journal) when completing significant work, and keep doc claims synchronized with code.
