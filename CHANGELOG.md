# Changelog

All notable changes to Maestriss are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). Maestriss follows Semantic Versioning 2.0.0 with the pre-1.0 interpretation defined in `Documentation/Reference/Human/12 - Development Workflow and Engineering Standards.md` (Versioning and Release Policy). Studio, the Native Runner, the Automa exporter, participant drivers, and the Engineering Library ship as one Maestriss release train under one canonical version, owned by the root `package.json`.

Ordinary development accumulates under `Unreleased`. Entries move into a dated release section only when a release is approved by a human, and Git tags (`vMAJOR.MINOR.PATCH`) are created only for approved release commits.

## [Unreleased]

### Added

- First direct API execution path for Council runs (OpenAI Responses API). A Council run can now execute every Seat through the OpenAI API instead of the browser, without changing prompt composition. A narrowly scoped execution seam distinguishes transport from Mind identity: the existing `CouncilAskFn(mindId, prompt)` is retained, and a new `CouncilExecutionTarget` (`mindId`, `providerFamily`, `model?`, `transport: 'browser' | 'api'`) plus an optional `runCouncil` `executionOverride` force every Seat to one target while the configured chain is not walked. Composition is transport-independent: cognitive stats still resolve from the Seat's configured Preferred Mind, and the byte-identical composed prompt (Calling framing, cognitive guidance, maxResponseChars guidance, Memory/context, original request) is what reaches the API — no special API prompt. The shared transport (`runner/src/openaiTransport.ts`, official `openai` Node SDK) owns client construction, the Responses API call, `output_text` extraction, and structured failure classification, and is shared with the smoke test; `createOpenAiCouncilAsk` adapts it to the Council ask contract. A run-level CLI flag `--mind openai-api` (`npm run dev -- council run --doctrine dream-lab --prompt-file <path> --mind openai-api --verbose`, model `gpt-4o-mini`) runs an all-OpenAI Council: every Seat keeps its canonical Calling, cognitive configuration, maxResponseChars, Memory/context policy, speaking order, and Doctrine choreography — only the executing target is overridden, and canonical Calling affinities, Suggested Minds, Doctrine Preferred Minds, and saved configurations are never mutated. The structured result distinguishes configured Preferred Mind, run override, and actual executing target (`CouncilSeatProviderSelection.executionOverride`/`executedTarget`; `CouncilRunResult.executionOverride`). Forced API mode bypasses the normal browser fallback chain: an OpenAI failure applies the Seat's existing failure policy and is never silently redirected to a browser provider. It integrates with run-scoped availability — only the OpenAI API Mind is preflighted (no unrelated browser providers, and no paid dummy generation: the first real Seat ask establishes dynamic availability), a missing `OPENAI_API_KEY` fails clearly before any Seat, and structured OpenAI failures map deliberately into Council semantics: authentication → `session-unavailable`, billing/quota → `usage-limit`, rate limit → `rate-limited`, network/availability → `provider-unavailable` (all run-unavailable, circuit-breaking the API Mind for the run), while a malformed response → `malformed-response` (a non-availability failure owned by the Seat failure policy, never disguised as provider unavailability). API responses participate fully in existing diagnostics (`responseChars`, `responseTargetExceeded`, over-target output preserved and non-failing). The isolated live smoke test (`npm run test:openai-api`, exact-response validation retained) and a new deterministic browser-free/network-free transport suite (`npm run test:openai-transport`, injected caller) share the transport code. `maxResponseChars` stays a prompt-side behavioral target (not mapped to `max_output_tokens`); native inference-parameter mapping remains future work. `OPENAI_API_KEY` is read from the environment and never printed, logged, or stored by Maestriss; API billing is separate from any ChatGPT subscription. Suites grew to: council-execution 113, council-cli 69, plus openai-transport 27.
- Run-scoped provider availability memory for Council runs: provider unavailability is treated as a routine operating condition, not an edge case, so availability learned during a run is remembered for the remainder of that run and never persisted (`runner/src/councilAvailability.ts` — a fresh registry per run holding `unknown`/`ready`/`unavailable` states with structured evidence: reason, message, source `preflight`/`readiness`/`ask`, and a run-relative first-detected timestamp). When enabled, one preflight readiness pass at Council start classifies every Mind that can appear anywhere in the Formation's configured chains (using the existing Runner readiness inspection — no duplicated provider logic; the existing taxonomy already identifies the Claude/Perplexity Cloudflare "verify you are human" loop as `security-verification`, the Copilot tenant block as `provider-blocked`, and login/plan gates), and preflight never sends a prompt. Each seat derives an effective execution chain at runtime as its configured chain minus the run's unavailable Minds (configured order preserved); the saved Council Configuration, its `providerFallbacks`, and each seat's Preferred Mind are never mutated. A Mind established unavailable — at preflight, by a readiness gate, or by a fallback-eligible structured ask failure — is skipped on every later seat with no fresh check, no ask, and no retry budget consumed (a run-scoped circuit breaker), while `unknown` Minds remain executable (the first real ask establishes their state) and a Mind that passes preflight may still transition `ready → unavailable` during an ask. Availability filtering happens strictly before the existing per-seat failure policy: generic timeouts, refusals, composition errors, and any non-availability failure never mark a Mind unavailable, and `providerUnavailableAskReasons` remains the single source of truth for ask-phase circuit-breaking (no second list). Structured state and events (`providerAvailability` on the run result; `CouncilPreflightStart`/`CouncilProviderAvailabilityEvent`/`CouncilPreflightComplete`/`CouncilProviderSkippedEvent`, `effectiveChain` on `CouncilSeatBegin` and `CouncilSeatProviderSelection`) feed both the CLI and a future Studio live UI (presentation deliberately undesigned). The Council CLI opens `council run` with a live "CHECKING MINDS..." check and a grouped READY / UNAVAILABLE / UNKNOWN availability summary with an available-of-total count, shows the configured-versus-effective chain per seat (keeping the Preferred Mind visible even when unavailable), reports a Mind becoming unavailable mid-run, and prints a concise run-scoped skip line on later seats instead of repeating the full explanation. Suites grew to: council-execution 99, council-cli 63 (council 401, cognitive-stats 105 unchanged).
- Canonical per-seat response-length target: `CouncilStage.maxResponseChars` (explicit integers 32-8192; absent seats resolve to the shared `defaultMaxResponseChars` of 1024 via one pure resolver, so existing configurations remain valid and all built-in Doctrine seats inherit the default). A prompt-side requested ceiling only — not max_tokens, not a native provider parameter, not truncation: every seat prompt carries one deterministic instruction ("Keep your complete response within approximately N characters. Prioritize the most important content if space is limited." — raw integer, characters, never tokens) as its own layer between cognitive guidance and the output instruction, actual provider output is always preserved, and over-target responses still pass with the overrun recorded structurally (`responseChars`, `responseTargetExceeded` on seat records; `resolvedMaxResponseChars` in composition metadata for the future Studio UI). Independent from Voice in both directions (asserted); compose-once fallback guarantees every fallback Mind receives the byte-identical prompt with the same target. The Council CLI shows the resolved target per seat (Max response length), reports actual response length against it, and warns with the exact overage when exceeded; verbose diagnostics expose the resolved value. Suites grew to: council 401, cognitive-stats 105, council-execution 80, council-cli 55.
- Deterministic provider fallback for Council seats, under the canonical seat model (a Seat is a stable positional identity owning its Calling as bound cognitive duty, its preferred Mind as bound AI, its cognitive overrides, and its fallback preferences; the same Calling may occupy multiple seats with different Minds and overrides; the preferred Mind is bound seat state but not guaranteed to be the executing provider): `CouncilStage.provider` (the preferred Mind) plus an additive `providerFallbacks` list of ordered alternate Minds forming one deterministic effective chain (`effectiveProviderChain`, at most five total choices; validation rejects unknown ids, duplicates anywhere in the chain, and oversized chains without silent correction). Each Calling's `providerAffinity` is now its canonical five-provider ranking, and Doctrine factories derive every built-in seat's role-sensitive fallback chain from it (assigned provider first, ranking minus that provider), leaving `providerFallbacks` as the per-seat customization point. Fallback is strictly availability-scoped: any non-ready readiness status, or an ask failure in the explicit `providerUnavailableAskReasons` taxonomy (account/plan blocks, login/session problems, usage/rate limits, capacity errors) — composition errors, timeouts, refusals, and other execution failures are never disguised. The seat prompt is composed exactly once from the preferred provider's seat identity and reused byte-identical by every chain provider; availability rejections consume no retry budget, normal retry-once/skip-and-record/halt semantics govern the executing provider, an availability failure on the executing provider advances the chain rather than being retried, and an exhausted chain resolves via the failure policy as `provider-unavailable` (halting under halt/retry-once). Structured selection state (`CouncilSeatProviderSelection` with preferred provider, chain, ordered rejections and reasons, executing provider, and exhaustion; `CouncilSeatBegin` and `CouncilProviderRejectedEvent` events) feeds both the CLI and a future Studio live UI (presentation deliberately undesigned). The Council CLI shows the provider chain per seat, each unavailability with its structured reason, fallback transitions with chain position and a seat-unchanged notice, executed-versus-preferred provider on completion, a PROVIDER CHAIN EXHAUSTED block listing every attempted provider, and (verbose) an FNV-1a prompt identity proving exact prompt reuse. Execution suite grew to 72 assertions, CLI suite to 51, contract suite to 388.
- Live Council CLI observability overhaul: `council run` now renders a full operator view from structured execution records — a run header (Doctrine, purpose, party size, request size, start time, ordered Formation), per-seat progress blocks (Calling, AI, stage id, input policy with a practical explanation, all six resolved cognitive stats with numeric level and display-only descriptor, context-flow diagnostics distinguishing input-policy eligibility, Memory selection, budget inclusion, Memory-excluded versus budget-omitted contributions, and truncation with original size and applied cap, prompt length, Calling flavour source, and failure policy), a 10-second elapsed heartbeat while an ask is in flight, explicit retry/skip/halt decision output (retry states that the same exact prompt is re-sent and no contribution was recorded), per-seat elapsed time and response length, a running pass/skip/fail progress summary, and a structured final summary (halted-seat detail on FAIL, skipped-seat list on PARTIAL, Crown Council honesty note retained). `--verbose` adds per-seat composition diagnostics (contribution ids at every pipeline stage, prompt budget usage, per-contribution cap, resolved cognitive instruction lines) plus the exact provider-facing prompt; normal mode never prints the prompt but always prints the full extracted response. Rendering lives in `runner/src/councilCliFormat.ts` and consumes new additive engine observability metadata (`CouncilSeatComposition`, per-seat `elapsedMs`/`responseChars`, an injectable clock, and a per-attempt `onSeatAttempt` callback in `runner/src/councilExecution.ts`; `effectivePerContributionChars` on composed prompts) with execution semantics unchanged. Display-only cognitive level descriptors live in `shared/council/cognitiveStatDescriptors.ts` (never provider-facing). Covered by a new deterministic suite, `npm run test:council-cli` (51 browser-free assertions); the execution suite grew to 72.
- Doctrine input-policy × Memory coherence audit (`npm run audit:doctrine-memory`, `runner/src/doctrineMemoryAudit.ts`): a deterministic, behavior-free diagnostic that simulates every canonical Doctrine through the real shared composition pipeline and reports, per seat, the input policy, resolved Memory level and its source layer, input-policy-eligible contribution ids, Memory-exposed ids, and a coherence assessment (including Crown Council independence verification). Built for reviewing Formation choreography against Memory intent before tuning; no browser, Runner service, or network.
- Canonical six cognitive stats (`shared/council/cognitiveStats.ts`, `shared/council/cognitiveGuidance.ts`): Temperament (Precise/Imaginative), Voice (Terse/Expansive), Conviction (Adaptable/Resolute), Dissent (Cooperative/Adversarial), Depth (Swift/Exhaustive), and Memory (Isolated/Full Council), each stored as one of ten deterministic levels 0–9 with 5 as canonical neutral. These are Maestriss behavioral dimensions expressed through deterministic provider-facing instructions and mechanical context exposure — never provider API parameters (no temperature, top_p, or reasoning-token claims; a future API-backed execution layer may additionally map compatible stats natively). One pure resolution function applies the canonical precedence seat override (`CouncilStage.cognitiveOverrides`, closed-object validated) > Calling default (approved partial characterizations on all sixteen Callings) > provider default (approved complete product calibrations for all nine providers, Google deliberately neutral; product defaults, not provider psychology) > system neutral, with Suggested AI never participating. Prompt composition renders a sparse "Cognitive guidance:" section from a fixed source-checked ten-level instruction catalog — neutral stats emit nothing, raw stat numbers are never sent, and guidance stays a separate layer from Calling flavour text. Memory acts mechanically: input policy stays authoritative for eligibility and original-request inclusion, Memory only narrows exposure (0 none; 1–3 newest-one at quarter/half/normal cap; 4/6/7/8 newest 2/3/4/6; 5 preserves input-policy behavior exactly; 9 all eligible within budgets; independent-round always isolated). Approved choreography seat overrides: Dream Lab first Wild Mage (temperament 9), Dream Lab second Wild Mage (temperament 9, memory 9), Trial by Fire second Saboteur (memory 9, dissent 9), Oracle's Table second Oracle (memory 9, depth 9), Socratic Circle second Sage (memory 9, depth 9), and every Crown Council Magistrate (memory 0). Composition now exposes `resolvedCognitiveStats`, `eligibleContributionIds`, and `memorySelectedContributionIds` diagnostics. Covered by a new deterministic suite, `npm run test:cognitive-stats` (100 browser-free assertions).
- Live Doctrine execution from the Runner CLI: `npm run dev -- council run --doctrine <id> (--prompt "text" | --prompt-file path) [--size N] [--verbose]` builds, validates, and executes a built-in Doctrine's Council Configuration seat by seat in Formation order over the existing `/ask` lifecycle (drivers, extraction, readiness, cancellation, and timeouts reused — never duplicated). The engine (`runner/src/councilExecution.ts`) follows the baton injected-ask architecture; every seat prompt is composed by the shared pipeline against real prior contributions per the seat's input policy and preserved verbatim on the in-memory seat record (`--verbose` prints it before sending). Failure policies execute as configured (halt; retry-once re-sending the same exact prompt and halting if the retry fails; skip-and-record forwarding nothing from the failed seat); runs classify as PASS / PARTIAL / FAIL; the reported final contribution is simply the last successful seat output; Crown Council runs explicitly note that vote aggregation is not yet implemented. Run state is in-memory only (persisted Council Records are the next slice) and Studio does not yet trigger execution. Covered by a new deterministic suite, `npm run test:council-execution` (72 browser-free assertions).
- Suggested AI per Calling: `CallingDefinition.suggestedProvider` records each Calling's single default best-fit provider (distinct from the broader ordered `providerAffinity` list; a soft recommendation that never invalidates other assignments and is never substituted for a Formation's assigned provider), displayed in the Calling Grimoire as "Suggested AI: <provider>" or "Varies by Council", resolved entirely from the shared registries.
- Sixteen built-in Doctrines on the Formation/seat model (ordered seats, each assigning a Calling and a provider; duplicate Callings and duplicate providers are valid; provider affinity is a soft hint only): Realm Summit, Dream Lab, Crucible, Imperial Court, Crown Council, Arcane Expedition, Scholar's Conclave, Grand Academy, Decision Chamber, Trial by Fire, War Room, Workshop, Oracle's Table, Creation Chamber, Socratic Circle, and Grand Campaign — all producing schema-valid, deterministic Council Configurations with expert-designed v1 Formations intended for tuning after live runs. Crown Council (the one size-configurable Doctrine, 2–12 seats, default 4) seats equal Magistrate peers deliberating independently in one shared round with providers rotated in registry order — equal deliberation is a Doctrine mechanic, not a Calling — with vote casting/tallying and any non-voting synthesis stage documented as future execution work; `CouncilDoctrine` carries a `defaultSize` field; Doctrine names carry no leading "The".
- Canonical sixteen-Calling library with approved provider-facing flavour text: Lantern Bearer, Inquisitor, Rival, Wild Mage, Magistrate, Royal Scribe, Saboteur (defensive failure-mode analysis), Empath, Alchemist, Cartographer, Oracle, Sage, Pathfinder, Archivist, Quartermaster, and Architect, each with practical titles, UI descriptions, default output policies, position hints, and soft provider affinities. The Calling Grimoire enumerates all sixteen automatically with per-Calling icons.
- Council-level Calling flavour overrides: `CouncilConfiguration` carries an optional compact `callingFlavourOverrides` record (explicitly customized Callings only; canonical text is never duplicated into configurations), validated against the Calling library and resolved by prompt composition ahead of canonical flavour text, so saved configurations reproduce customized behavior anywhere without depending on browser-local state. Doctrine factories accept overrides as a build option, and the Calling Grimoire projects local edits into the portable shape (shared converter + copy-as-JSON).
- Calling flavour-text library and Studio Calling Grimoire editor: the provider-facing Calling framing lives in the canonical `shared/council/callingFlavourText.ts`, and a Studio page lists every Calling (fantasy title with the practical title always visible), edits its flavour text with a live provider-facing preview, unsaved-change indication, and revert-to-canonical, persisting edits as versioned local-browser overrides that never modify the canonical source. Execution reads only a configuration's embedded `callingFlavourOverrides`, never browser state.
- Shared Council orchestration contract (`shared/council/`): canonical provider registry adopting the runner's execution-verified identities (the runner roster re-exports it with no behavioral change), the versioned Council Configuration schema (`schemaVersion: 1`) with a deterministic validator, council rules, behavioral variables, input/output/failure-policy vocabularies, prompt budgets, and a deterministic prompt-composition pipeline — all covered by the browser-free `npm run test:council` contract suite.
- Dedicated Perplexity response-filtering module (`runner/src/drivers/perplexityFiltering.ts`) with a deterministic assertion suite (`npm run test:perplexity-filter`), completing dedicated filter coverage for all nine providers.
- Sequential multi-provider baton test (`npm run dev -- baton-test [--seed <value>] [--skip-unavailable]`): sends a deterministic seed through all nine participants in a fixed order over the normal `/ask` lifecycle, requiring each provider to return exactly the previous provider's actual extracted output plus its own token. Fails immediately on any wrong, stale, or unavailable-provider result; `--skip-unavailable` skips not-ready providers before their ask and reports `PARTIAL` instead of `PASS`. Orchestration logic lives in `runner/src/batonTest.ts` with a browser-free deterministic assertion suite (`npm run test:baton`).

### Changed

- Suggested Mind / affinity alignment: it is now a canonical invariant that every Calling's affinity ranking leads with its Suggested Mind (`providerAffinity[0] === suggestedProvider`), asserted for all sixteen Callings, so there is exactly one canonical answer to which Mind best suits a Calling — seat-level divergence belongs to a seat's Preferred Mind, never to Calling metadata. Per explicit product decisions: Empath's default Mind is Reka (Reka → Claude → ChatGPT → Gemini → Copilot), Cartographer aligned to Gemini-first (Gemini → Claude → ChatGPT → DeepSeek → Grok), and Pathfinder aligned to Perplexity-first (Perplexity → Grok → Gemini → Google → ChatGPT); the former deliberate Cartographer/Pathfinder divergences are retired. Doctrine seat assignments are unchanged — a Doctrine remains free to bind any preferred Mind to any seat.
- Council variable migration to cognitive stats: Voice supersedes the `responseLength` variable, and the ten-level cognitive Dissent stat supersedes the three-level `dissent` variable (collaborative/balanced/adversarial intent maps onto levels 1/5/9, now carried by Calling cognitive defaults and the approved seat overrides rather than a configuration-global variable). Both legacy variables, their prompt instruction tables, and the Calling/Doctrine `dissent` defaults were removed cleanly — verified safe because no Council Scroll persistence has shipped — so `CouncilVariables` now carries exactly `roleIntensity` (Calling intensity, still selecting full flavour text versus the light one-liner, independent of cognitive stats) and `inputMode` (orchestration semantics, deliberately not collapsed into Memory). The locked full-prompt council snapshot was intentionally re-locked with the cognitive-guidance section, and refusal-safety screening now covers every cognitive instruction sentence.
- Canonical Council vocabulary migration: presets are now Doctrines, roles are now Callings, and the Doctrine → Formation → Seat → Council model is the product language across shared modules, Runner, Studio, tests, and documentation. Calling renames: Master of Questions → Sage, Scout → Pathfinder, Smith → Architect; the Councillor (General Deliberator) role was removed — equal deliberation is a Doctrine mechanic, not a Calling. Doctrine renames: Concord → Realm Summit, Idea Forge → Dream Lab, Editorial Court → Imperial Court, Council of X → Crown Council, Academy → Grand Academy, Creative Studio → Creation Chamber, Campaign → Grand Campaign. Shared modules renamed to `callings.ts`, `callingFlavourText.ts`, and `doctrines.ts`; schema fields renamed (`calling`, `doctrineId`, `callingFlavourOverrides` — performed before any persisted artifacts exist); Studio's Role Grimoire became the Calling Grimoire (navigation key `callings`). Deliberately retained compatibility identifiers: the `maestriss.roleFlavourOverrides` localStorage key, parser-level legacy calling-id migration (old ids and stored Councillor overrides migrate or drop on parse), and the internal `roleIntensity` variable name. Dream Lab seat 5 changed from Rival/Gemini to Pathfinder/Gemini during Formation review.
- Runner build output now emits under `dist/runner/src` and `dist/shared` because the runner compiles the shared council module; `npm start` points at the new entry path, and the runner CLI resolves its package root by walking up to the nearest `package.json` instead of assuming a fixed compiled depth.

### Removed

- Dormant graph-based workflow scaffolding: the unused `WorkflowDefinition`/`WorkflowNode`/`WorkflowEdge` types and `runner/workflows/google-chatgpt.workflow.json`, retired in favor of the shared Council Configuration model after a repository-wide reference check confirmed no consumer.

### Fixed

- Grok signed-out plan/upsell block is no longer accepted as a successful Council contribution (first live Dream Lab run, 2026-07-14: "Sign up to continue seamlessly with Grok's full power" won candidate selection, passed the seat, and entered Council context). Account/plan gating chrome now classifies as a structured availability failure (`grok-account-or-plan-block`, length-bounded phrase matching so long real answers quoting such wording are unaffected) at both the terminal-error and candidate-evaluation layers in `runner/src/drivers/grokFiltering.ts`, and the category is fallback-eligible so the seat's provider chain advances with the identical composed prompt. Fixtures added to `npm run test:grok-filter`; an end-to-end fallback assertion added to the execution suite.
- ChatGPT completion detection no longer times out on long completed responses (same live run: a visibly finished ~8,000-character answer was rejected every poll because the 1200px height cap classified the real assistant message — and its markdown and article containers — as a page/transcript parent, so `latestAssistantText` stayed empty for the full 120s). Provider-semantic single-message anchors (`[data-message-author-role="assistant"]` and the assistant conversation-turn testid) are now exempt from the size-proxy rejection in `runner/src/drivers/chatgptFiltering.ts`; the cap still guards broad fallback selectors, and all other rejection rules (prompt-only, shell text, smaller-valid-child) still apply to semantic nodes. Live-shaped fixtures added to `npm run test:chatgpt-filter`.
- Structured driver failure categories now survive the server's ask-failure path (`runner/src/askFailureReason.ts`): a leading kebab-case category token thrown by a driver (e.g. `grok-capacity-error: ...`) is recovered into the normalized response's `reason` field instead of being flattened into free text — previously no ask-phase availability category could ever match the Council fallback taxonomy live. Free-text failure messages are unchanged.

- DeepSeek response detection no longer rejects valid answers in narrow/responsive layouts (live baton stall: the correct answer at x=20 was rejected as `left-sidebar-region` by an absolute `left < 250` cutoff for over 100 seconds). Geometry rules are now composer-relative — only candidates entirely left of the composer column are rejected geometrically, the transcript width cap is measured against the composer width, and structural sidebar ancestry rejection is unchanged — with the rules extracted to `deepSeekGeometryRejectionReason` in `runner/src/drivers/deepseekFiltering.ts` and geometry assertions (including the exact live fixture) added to `npm run test:deepseek-filter`.
- Reka response extraction no longer returns a transcript-level parent containing the submitted prompt plus the answer (live baton false positive with long prompts; the prompt-only guard capped at 120+80 characters). Detection now prefers Reka's semantic assistant-answer containers (`div.prose.prose-chat` markdown nodes) over broad page-wide containers, structurally rejects the `justify-end` user-prompt bubble, and rejects any candidate containing the full submitted prompt, with candidate selection extracted into testable functions in `runner/src/drivers/rekaFiltering.ts` and response-selection assertions added to `npm run test:reka-filter`.
- Copilot readiness now classifies the Microsoft 365 `/chat/blocked` page ("Copilot Chat isn't available") as an explicit `provider-blocked` status with a diagnostic note, instead of the generic `unknown`/no-composer result.
- Reka submission no longer hard-fails on long multiline prompts (live baton failure: expanded composer pushed the send button outside the geometric composer box, zero candidates, and the failure fired before any keyboard fallback). Submit-control discovery now uses structural evidence first — the send button shares the composer's `<form>` (lucide-send icon) — with geometry as supporting evidence only (`runner/src/drivers/rekaSubmitTargets.ts`), and a missing clickable target no longer prevents verified keyboard submit strategies from running. Deterministic submit-target assertions were added to `npm run test:reka-filter`, including a fixture shaped on the saved failure DOM.
- Perplexity response extraction no longer returns a transcript-level parent containing the submitted prompt plus the answer (live baton false positive with long prompts). Detection now prefers Perplexity's stable semantic answer containers (`[id^="markdown-content-"]`, prose/markdown nodes) over `main`/`article` fallbacks, structurally rejects the `group/query` prompt container, and rejects any candidate containing the full submitted prompt.

## [0.2.1] - 2026-07-11

### Added

- Dedicated ChatGPT response-filtering module (`runner/src/drivers/chatgptFiltering.ts`) with a deterministic assertion suite (`npm run test:chatgpt-filter`) covering response filtering, cleaning, geometry-based parent rejection, submission evidence, and browser-script validity.
- ChatGPT submission verification: click, keyboard Enter, and event-based Enter strategies must produce accepted-submission evidence (composer cleared, prompt visible as user message, stop control visible, or assistant response changed) before the driver reports success.
- Versioning and release system: canonical version ownership in the root `package.json`, this changelog, a read-only version verifier (`npm run verify:version`) with deterministic fixture-based assertions (`npm run test:version-verifier`), and a local-only `npm run dev -- version` runner CLI command.

### Changed

- AI onboarding redesigned into role-specific generated pairs: `Web_AI_Prompt.md`/`Web_AI_Bootstrap.md` for high-level project AIs and `VSC_AI_Prompt.md`/`VSC_AI_Bootstrap.md` for repository-attached engineer AIs.

### Fixed

- ChatGPT browser-evaluated detection scripts restored to valid standalone JavaScript (live `page.evaluate` SyntaxError during completion detection and extraction) and guarded against transpiler `__name` injection, with deterministic script-validity assertions.
- Claude response filtering no longer rejects a clean, short, visible answer as a parent container when its raw text already equals its cleaned text (live `Claude OK` false-negative), with a deterministic regression case for the exact live geometry.
- Gemini response detection now prefers observed structural response containers over brittle sidebar/history geometry, preserving live `Gemini OK` answers in the mobile/narrow layout while still rejecting navigation text.
- Live exact-answer validation passed for ChatGPT, Claude, Gemini, Google AI Mode, DeepSeek, Grok, Perplexity, and Reka. Copilot was externally unavailable on a Microsoft 365 blocked page and is not counted as a live pass.

## [0.2.0] - 2026-07-10

v0.2.0 "Documented Foundation" — first tagged release (`v0.2.0`, commit `d6eb1a8`). Earlier development (initial Studio skeleton through native-runner introduction) was untagged; see `Documentation/Handoffs/2026-07-10 - Native Runner Foundation Milestone.md` for the full milestone record.

### Added

- Native Runner: Node + TypeScript + Playwright engine with HTTP server (`/health`, `/providers/status`, `/ask`, `/inspect`, `/cancel-all`, `/chain`, `/run-random`) and CLI over one core.
- Nine participant drivers (ChatGPT, Claude, DeepSeek, Gemini, Google AI Mode, Grok, Copilot, Perplexity, Reka) with candidate-based extraction and logged rejection reasons.
- Two browser execution modes: persistent Playwright profile and CDP attach to an external Chrome (`runner/restart-runner.ps1`).
- Seven deterministic provider filter regression suites (claude, gemini, google, grok, copilot, deepseek, reka).
- Diagnostics vocabulary, failure artifacts in `runner/debug/`, and `inspect` tooling.
- Maestriss Studio (nine pages) with project JSON, prompt-pack, session-transcript, and Automa workflow exports.
- Documentation knowledge system: 16-document Reference library in Human and AI editions, Knowledge System Guide, handoffs, and reviews, plus a docs-vs-code reconciliation audit.
