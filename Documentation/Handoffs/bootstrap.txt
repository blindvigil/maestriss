MAESTRISS — AI PROJECT BOOTSTRAP PROMPT

You are joining the Maestriss project as an engineering collaborator.

Maestriss is a multi-AI orchestration platform. It coordinates independent AI systems through their real web interfaces, treating each AI provider as a participant in structured workflows, chains, roundtables, reviews, and synthesis processes.

The currently supported participant roster is:

- ChatGPT
- Claude
- DeepSeek
- Gemini
- Google AI Mode
- Grok
- Microsoft Copilot
- Perplexity
- Reka

The project currently contains three related but distinct implementation surfaces:

1. Maestriss Studio
   A React, TypeScript, and Vite application intended to become the main visual configuration and orchestration interface.

2. The Native Runner
   A Node, TypeScript, and Playwright automation service that operates the live AI provider tabs through browser automation.

3. The Automa Exporter
   A Studio-side exporter that generates Automa-compatible workflow artifacts as an alternative execution path.

The Native Runner is presently the operational execution engine. Studio-to-runner live integration is future work and must not be assumed to exist unless verified in the current code.

CURRENT ARCHITECTURAL BASELINE
==================================================

The Native Runner follows this request lifecycle:

Resolve participant
→ acquire or reuse browser tab
→ bring the participant tab to the front
→ handle any manual security-verification gate
→ waitForReady()
→ pastePrompt()
→ verify the pasted prompt
→ submitPrompt()
→ verify submission where supported
→ waitForCompletion()
→ extractResponse()
→ clean and normalize the response
→ return the result

The lifecycle is shared, but provider behavior is intentionally specialized.

Provider-specific behavior belongs in provider drivers and filtering modules.

Shared orchestration, queueing, browser management, HTTP handling, tab reuse, and lifecycle coordination belong in shared runner infrastructure.

Do not force provider-specific behavior into a universal abstraction merely for aesthetic consistency.

==================================================
BROWSER EXECUTION MODES
==================================================

Maestriss supports two browser execution modes:

1. Playwright persistent-profile mode
   - Default when the runner launches its own browser context.
   - Uses Playwright Chromium.
   - Uses the profile:
     runner/.user-data

2. Chrome DevTools Protocol mode
   - Connects to an externally started Chrome instance.
   - Commonly uses:
     http://127.0.0.1:9222
   - Uses the profile:
     %LOCALAPPDATA%\MaestrissChromeProfile

The two modes use separate browser profiles. Authentication and provider sessions do not automatically transfer between them.

CDP mode is commonly used for visible, interactive development and “watch it cook” operation.

Do not describe the two modes as interchangeable without noting their separate profiles and session state.

==================================================

Do not assume all providers have equal implementation maturity.

At the current milestone:

- Claude, Gemini, Google, Grok, Copilot, DeepSeek, and Reka have provider-specific filtering modules and regression assertions.
- ChatGPT does not yet have a dedicated filtering module or filter assertion suite.
- Perplexity has meaningful inline filtering and candidate logic but does not yet have a separate filtering module or regression suite.
- Some providers still contain known technical debt or provider-specific heuristics.
- Submission verification maturity varies by provider.
- Live provider behavior may change without warning because the project automates third-party web interfaces.

Verify current maturity against the code before repeating these statements, since later commits may have changed them.

==================================================
ENGINEERING PRINCIPLES
==================================================

Follow these principles:

- Reliability over cleverness.
- Evidence over assumptions.
- Verify visible state changes rather than relying on arbitrary delays.
- Paste must be verified.
- Submission should be verified.
- Completion should be based on stable response state and active generation controls, not fixed sleeps.
- Response extraction should prefer the smallest valid assistant-answer candidate.
- Prompt echoes, sidebars, navigation, status labels, suggestions, and page chrome must not be returned as answers.
- Geometry is useful evidence, but not universal truth.
- Provider-specific geometry rules are acceptable when empirically justified and documented.
- Static text such as “Stopped generating” is not the same as an active stop button.
- Provider errors, rate limits, and capacity messages are failures or provider state, not valid answers.
- Provider-specific quirks should remain isolated.
- Working heuristics should not be removed merely because they are inelegant; replace them only with evidence-backed improvements.
- Every difficult bug should ideally produce a regression test, stronger diagnostics, or both.
- Diagnostics are first-class product features.
- Documentation should be updated when implementation changes invalidate it.
- Stable systems should not be redesigned without a clear requirement and evidence that redesign is necessary.
